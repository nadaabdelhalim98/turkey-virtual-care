e-Nabız Clinical Alert Layer — API Specification v1.0
This document specifies the API design for the e-Nabız Clinical Alert Layer — a proposed integration between Turkey's national personal health record platform (e-Nabız), the MHRS appointment system, and family physician practice management systems (PVS). This is a design specification, not a live API. All endpoints are proposed additions to the existing e-Nabız infrastructure. Base URL would be: https://api.enabiz.gov.tr/v2/clinical-alerts
01 · Alert Threshold Configuration
Alert Level Systolic Threshold Diastolic Threshold Consecutive Days Action Triggered
WARNING ≥ 150 mmHg ≥ 90 mmHg 3 days KIM message to GP — monitoring flag
CRITICAL ≥ 160 mmHg ≥ 95 mmHg 3 days KIM alert + MHRS slot suggestion + GP notification
HYPERTENSIVE URGENCY ≥ 180 mmHg ≥ 110 mmHg 1 reading Immediate GP alert + patient call prompt + emergency escalation path
CONTROLLED < 140 mmHg < 90 mmHg — Weekly summary to GP. No action required.

03 · API Endpoints
POST
/readings
Submit a new BP reading for a registered patient
ParameterTypeDescription
tc_id requiredstringPatient Turkish ID (hashed — PII not stored in plaintext)
systolic requiredintegerSystolic BP in mmHg (60–300 valid range)
diastolic requiredintegerDiastolic BP in mmHg (40–200 valid range)
pulseintegerHeart rate in bpm (optional)
measured_at requiredISO 8601Timestamp of measurement
device_idstringUSBS-registered device identifier (optional)
sourceenummanual | wearable | pharmacy_kiosk
// Request body example { "tc_id": "hashed_patient_id_abc123", "systolic": 178, "diastolic": 104, "pulse": 82, "measured_at": "2026-06-07T07:23:00+03:00", "source": "manual" } // Response — alert triggered { "reading_id": "rdg_20260607_0723_abc", "status": "recorded", "alert_triggered": true, "alert_level": "CRITICAL", "alert_id": "alt_20260607_001", "gp_notified": true, "notification_channel": "KIM" }
GET
/alerts
Retrieve active alerts for a registered GP's patient panel
ParameterTypeDescription
gp_id requiredstringGP registration number (MoH licensed HCP)
statusenumactive | acknowledged | resolved | all (default: active)
levelenumWARNING | CRITICAL | URGENCY (default: all)
from_dateISO 8601Filter alerts from date
limitintegerMax results (default: 20, max: 100)
// Response { "alerts": [ { "alert_id": "alt_20260607_001", "patient_name": "Fatma Y.", "patient_age": 62, "level": "CRITICAL", "latest_bp": { "systolic": 178, "diastolic": 104 }, "7day_avg": { "systolic": 172, "diastolic": 101 }, "consecutive_days_elevated": 4, "medications": ["Amlodipine 5mg", "Ramipril 5mg", "HCTZ 12.5mg"], "created_at": "2026-06-07T09:14:00+03:00", "status": "active", "suggested_action": "teleconsult_or_in_person" } ], "total": 2, "generated_at": "2026-06-07T10:00:00+03:00" }
POST
/alerts/{alert_id}/respond
Log GP response to an alert — triggers patient notification and MHRS action
ParameterTypeDescription
action requiredenumschedule_teleconsult | schedule_in_person | send_message | acknowledge | escalate_emergency
gp_notestringClinical note (stored in patient PVS record)
appointment_slotISO 8601Proposed appointment time (if scheduling)
message_to_patientstringPlain-language message sent to patient via e-Nabız notification

04 · Security & Compliance
// Authentication — OAuth 2.0 with MoH identity provider Authorization: Bearer {moh_issued_jwt_token} // All tokens are: // - Issued by MoH identity server // - Scoped to specific GP registration number // - Valid for 8 hours (refresh via /auth/refresh) // - Logged for KVKK audit trail // Data handling: // - TC identity numbers hashed (SHA-256) before storage // - All health data encrypted at rest (AES-256) // - All API calls over TLS 1.3 // - KVKK Article 6 compliant — health data processing // - Patient consent required before GP data sharing (stored in e-Nabız consent record) // - Full audit log retained for 10 years per MoH data retention rules
About This Specification

This API specification is a portfolio design artefact produced by Nada Abdelhalim as part of the Turkey Virtual Care Readiness project. It demonstrates how the e-Nabız Clinical Alert Layer innovation proposal could be technically implemented within Turkey's existing digital health regulatory framework (KVKK, USBS, MoH licensing requirements). It is not connected to the live e-Nabız system, which is operated by the Turkish Ministry of Health and requires institutional authorisation for API access. All patient data shown in the prototype is entirely simulated.
